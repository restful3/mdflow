# M4 â€” MCP Server (FastMCP) Implementation Plan

> **For agentic workers:** Implement task-by-task with TDD (failing test â†’ verify fail â†’ implement â†’ verify pass â†’ ruff â†’ commit). Steps use checkbox (`- [ ]`) syntax.

**Goal:** Expose the existing `ConversionService` over MCP via FastMCP, with 4 tools (`convert_file`, `convert_url`, `list_formats`, `get_cached`), a `mdflow-mcp` stdio entrypoint, and a `/mcp` Streamable HTTP mount on the FastAPI app. No conversion logic is added â€” tools wrap `service.convert`, `convert_from_url`, `registry.list_formats`, `cache.read`.

**Architecture:** A shared `build_registry(settings)` composition root (extracted from the lifespan, so HTTP and MCP never diverge) feeds a `Runtime` bundle that the tools close over. Async tools offload the synchronous service to `loop.run_in_executor(None, ...)`; a progress bridge marshals the converter's sync `progress(stage,pct)` callback to `ctx.report_progress` via `asyncio.run_coroutine_threadsafe` (the SSE handler's `call_soon_threadsafe` analogue). `MdflowError` â†’ `ToolError("[CODE] msg")`. GPU serialization is intentionally NOT applied (no `requires_gpu` converter is registered; M2b will revisit SSE+MCP together).

**Tech Stack:** Python 3.12, `fastmcp 3.3.1` (+ transitive `mcp 1.27.1`) â€” verified: `FastMCP.tool` decorator, `Context.report_progress(progress, total, message)` (async), `FastMCP.run(transport=...)`, `FastMCP.http_app(path=...)`, `from fastmcp import Client` (in-memory), `from fastmcp.exceptions import ToolError`. Prototype confirmed end-to-end (in-memory Client â†’ tool â†’ run_in_executor(service.convert) â†’ `result.data` dict).

## Important findings from planning (read before implementing)

- **In-memory Client result shape**: `result = await client.call_tool(name, args)` exposes `result.data` and `result.structured_content` (both the returned dict for a dict-returning tool). Tests assert on `result.data`.
- **`ctx` injection**: a tool param annotated `ctx: Context` (or default `None`) receives the FastMCP context. `report_progress` no-ops if the client didn't send a `progressToken`, so the bridge is always safe to call.
- **`ToolError`** lives at `fastmcp.exceptions.ToolError`. Raising it surfaces a clean MCP tool error to the client.
- **`run_coroutine_threadsafe` bridge** is fire-and-forget: the worker thread does not block on progress delivery (progress is best-effort; the conversion result is canonical). Verified harmless in the prototype.
- **http_app lifespan**: `FastMCP.http_app(path="/mcp")` returns a Starlette app with its own lifespan. Mounting on FastAPI requires combining lifespans (FastMCP docs pattern). Task 5 verifies the exact wiring; if it proves fiddly, ship stdio + tools first and treat the HTTP mount as the last slice of this same milestone.
- **GPU**: `service.convert` does lookup+run internally with no GPU gate. Since no `requires_gpu` converter is registered, this is correct today. Do NOT replicate the SSE gpu_semaphore dance here (YAGNI; documented in spec Â§1.3).
