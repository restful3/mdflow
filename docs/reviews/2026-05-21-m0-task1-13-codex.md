## 총평

M0 골격은 전반적으로 작고 명확하게 나뉘어 있고, URL fetch 합의안의 큰 방향도 코드와 테스트에 많이 반영되어 있습니다. 특히 SSRF 기본 차단, redirect별 재검증, fixed UA, size cap, request별 fetch metadata 분리 테스트는 의미 있게 작성되어 있습니다.

다만 Task 14 진입 전에 고치는 편이 좋은 차단급 문제가 3건 있습니다. 핵심은 "캐시 키가 변환 결과를 결정하는 모든 입력을 포함하지 않는다", "URL `Content-Type` hint가 실제 포맷 감지에 쓰이지 않는다", "일부 잘못된 URL이 구조화된 `MdflowError`가 아니라 `httpx.InvalidURL`로 새어 나온다"입니다.

## 차단

### 1. 캐시 키가 `filename_hint`/detected format을 무시해 같은 bytes가 다른 포맷으로 들어오면 잘못된 캐시를 반환합니다

위치:
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/service.py:56`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/service.py:68`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/cache.py:29`

문제:
- `ConversionService.convert()`가 `detect_format()`보다 먼저 `compute_cache_key(req.data, req.options)`를 계산합니다.
- 하지만 현재 변환 결과는 `req.data + req.options`만이 아니라 `filename_hint`가 만든 detected format에도 의존합니다.
- 같은 bytes라도 `a.txt`와 `a.csv`는 다른 converter behavior가 될 수 있는데, 두 번째 요청은 첫 번째 캐시를 그대로 반환합니다.

근거:
- PRD/합의안은 URL 문자열을 cache key에 넣지 말자는 뜻이지, 변환 결과를 바꾸는 format hint까지 무시하자는 뜻은 아닙니다.
- PRD §5는 포맷 감지가 변환 라우팅을 결정한다고 되어 있고, 현재 `TextConverter`는 `txt`와 `csv`에서 실제 출력이 달라집니다.

재현 예:

```python
data = b"hello\nworld\n"
service.convert(ConvertRequest(data=data, filename_hint="a.txt"))  # passthrough
service.convert(ConvertRequest(data=data, filename_hint="a.csv"))  # cache hit; csv table로 변환되지 않음
```

권고 조치:
- 포맷 감지를 캐시 조회 전에 수행하고, cache key의 canonical options 쪽에 `detected_format` 또는 `format`을 포함하세요.
- URL cache dedup 원칙은 유지할 수 있습니다. 같은 bytes + 같은 detected format + 같은 options이면 URL이 달라도 공유하고, 다른 hint로 다른 format이 되면 분리하면 됩니다.
- 회귀 테스트를 추가하세요: 같은 bytes를 `.txt` 후 `.csv`, `.csv` 후 `.txt` 순서로 요청했을 때 각각 올바른 출력이 나와야 합니다.

### 2. URL `Content-Type`이 `FetchResult`에는 저장되지만 포맷 감지에 사용되지 않습니다

위치:
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/url_fetch.py:224`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/url_pipeline.py:65`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/service.py:68`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/format_detect.py:94`

문제:
- URL fetch는 `content_type`을 수집하지만, `convert_from_url()`은 `filename_hint`만 `ConvertRequest`에 넘깁니다.
- `detect_format()`도 `data`와 `filename_hint`만 받습니다.
- 그래서 `Content-Type: text/plain`이고 URL path/Content-Disposition filename이 없는 정상 응답은 `FORMAT_DETECT_FAILED`가 됩니다. 합의안 §3.2 step 9의 "magic bytes -> Content-Type -> Content-Disposition -> URL path extension" 순서와 어긋납니다.

근거:
- 최종 합의안 §3.2 step 9는 `Content-Type`을 filename hint보다 앞선 hint로 쓰도록 명시합니다.
- PRD도 URL 입력에서 Content-Type/magic bytes 기반 라우팅을 전제로 합니다.

권고 조치:
- `ConvertRequest`와 `detect_format()`에 `content_type_hint`를 추가하거나, URL pipeline에서 Content-Type을 명시적인 format hint로 변환해 서비스에 전달하세요.
- `text/plain`은 libmagic 자동 판별에서는 제외한 현재 판단이 맞지만, HTTP 응답의 명시적 `Content-Type`으로 온 경우에는 `txt` hint로 취급하는 것이 합의안에 맞습니다.
- 테스트를 추가하세요: `https://example.com/` + `Content-Type: text/plain` + no filename path가 txt로 변환되어야 합니다. `text/csv`, `text/markdown`, OOXML MIME도 최소 1개씩 있으면 좋습니다.

### 3. 잘못된 port 등 일부 malformed URL이 `MdflowError(URL_INVALID)`로 정규화되지 않습니다

위치:
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/url_fetch.py:62`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/url_fetch.py:232`

문제:
- `validate_url()`은 `parsed.port`를 확인하지 않아서 `https://example.com:bad/path`가 통과합니다.
- 이후 httpx가 `httpx.InvalidURL`을 던지는데, 현재 catch는 `TimeoutException`과 `RequestError`만 처리합니다. `httpx.InvalidURL`은 `RequestError` 하위 타입이 아니어서 구조화된 `MdflowError`가 아니라 raw exception으로 새어 나옵니다.

근거:
- 합의안 §3.2 step 1은 URL 기본 검증 실패를 `URL_INVALID`로 거부한다고 명시합니다.
- Task 14 API에서 이 예외가 그대로 올라오면 표준 SSE/API error mapping 전에 500 계열로 새어 나갈 수 있습니다.

권고 조치:
- `validate_url()`에서 `parsed.port`를 한 번 접근해 `ValueError`를 `MdflowError(ErrorCode.URL_INVALID, ...)`로 감싸세요.
- `fetch_url()`에서도 `httpx.InvalidURL`은 `URL_INVALID`로 별도 catch하세요.
- `https://example.com:bad/path`, 공백 포함 host 같은 negative case를 `tests/test_url_fetch.py`에 추가하세요.

### 4. `ConvertRequest.fetch_metadata` 필드와 `url_pipeline`의 실제 인터페이스가 서로 다른 방향을 가리킵니다

문제:
- `ConvertRequest.fetch_metadata` 주석은 API layer가 URL metadata를 채운다고 되어 있지만, `ConversionService.convert()`는 이 필드를 전혀 사용하지 않습니다.
- 실제 URL pipeline은 `UrlConvertResponse(response, fetch)`로 sidecar를 반환합니다.

근거:
- 합의안 §3.7은 cache metadata와 request별 fetch metadata를 분리하되, 응답에서는 합성해야 한다고 했습니다.
- 지금 구조도 가능하지만, 두 인터페이스가 동시에 존재하면 Task 14에서 한쪽을 잘못 쓰기 쉽습니다.

권고 조치:
- 둘 중 하나로 정리하세요.
- 추천은 `ConvertRequest.fetch_metadata`를 제거하고, URL 경로는 `UrlConvertResponse`를 명시적으로 compose하는 방식입니다.
- 또는 서비스가 `fetch_metadata`를 받아 `ConvertResponse`/`Done.metadata`에 합성하게 하려면 `url_pipeline`도 그 경로를 쓰도록 맞추세요.

### 5. cache read/write I/O 및 오염된 cache entry가 `CACHE_IO_ERROR`로 분류되지 않습니다

위치:
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/cache.py:81`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/cache.py:89`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/service.py:58`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/service.py:97`

문제:
- `meta.json`이 깨져 있거나 읽기/쓰기 중 I/O 오류가 나면 `json.JSONDecodeError`, `OSError` 등이 그대로 전파됩니다.
- `ErrorCode.CACHE_IO_ERROR`는 정의되어 있지만 실제 cache 경로에서 쓰이지 않습니다.

근거:
- PRD §8.1은 `CACHE_IO_ERROR`를 표준 retryable error로 정의합니다.
- 리뷰 요청에서도 "service의 unsupported format/cache 오염"을 확인 포인트로 지정했습니다.

권고 조치:
- `ConversionService`에서 `cache.read()`/`cache.write()`를 감싸 `MdflowError(CACHE_IO_ERROR, ...)`로 정규화하거나, read 실패는 cache miss로 우회하고 warning metadata를 남기는 정책을 명시하세요.
- 깨진 `meta.json`, 읽기 권한 오류, write 실패를 테스트로 추가하세요.

### 6. cache atomic write의 tmp 경로가 sha마다 고정이라 동일 sha 동시 write에서 경합할 수 있습니다

위치:
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/cache.py:63`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/cache.py:77`

문제:
- 같은 sha에 대해 두 writer가 동시에 `write()`하면 둘 다 `.tmp-{sha}`를 사용합니다.
- 한 writer가 다른 writer의 tmp dir을 지우거나, `os.replace()` 직전에 tmp가 사라져 예외가 날 수 있습니다.

근거:
- plan R6는 같은 sha 동시 미스가 가능하되 deterministic이라 실질 영향이 작다고 했습니다. 하지만 현재 구현은 "마지막 replace 승" 수준이 아니라 tmp dir 충돌 예외가 날 수 있습니다.

권고 조치:
- `tempfile.mkdtemp(prefix=f".tmp-{sha}-", dir=self.root)` 같은 unique temp dir을 쓰고, 성공 시 `os.replace(tmp, entry)` 하세요.
- 실패 시 tmp cleanup을 `finally`에서 처리하면 됩니다.

### 7. URL fetch는 합의안의 temp-file streaming이 아니라 메모리 bytearray 누적입니다

위치:
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/url_fetch.py:206`
- `/media/restful3/data/workspace/mdflow/src/mdflow/core/url_fetch.py:220`

문제:
- 응답을 `bytearray`에 계속 누적한 뒤 `bytes(data)`로 한 번 더 복사합니다.
- 합의안 §3.2 step 7은 temp file streaming을 명시합니다.

근거:
- M0의 `ConversionService`가 bytes-only라 현재 구조가 단순하다는 장점은 있습니다.
- 다만 기본 URL cap이 200MB이면 fetch 단계에서만 최소 2배 메모리 복사가 생길 수 있고, 이후 converter도 bytes를 들고 갑니다.

권고 조치:
- Task 14 진입 자체를 막을 정도는 아니지만, M1 전에는 `FetchResult`가 temp path를 가질 수 있게 하거나 `SpooledTemporaryFile`/temp file 기반으로 바꾸는 편이 좋습니다.
- 단기적으로는 `MDFLOW_MAX_URL_INPUT_MB` 기본값을 더 낮추는 것도 운영 리스크를 줄입니다.
