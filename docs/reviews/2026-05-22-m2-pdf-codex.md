## Blockers

### 1. client disconnect 시 GPU semaphore가 in-flight GPU executor task보다 먼저 해제될 수 있습니다

위치:
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:216`
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:219`
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:220`
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:72`
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:75`
- `/home/restful3/workspace/mdflow/src/mdflow/runtime/concurrency.py:33`

문제:
- GPU branch는 `async with pool.gpu_lock(): async for chunk in _run_conversion_stream(...): yield chunk` 구조입니다.
- `_run_conversion_stream()`은 `started`를 yield한 뒤 executor task를 만들고, queue를 drain하며 progress/done/error를 yield합니다.
- 클라이언트가 executor task 시작 후 스트림을 끊으면 async generator가 닫히면서 `async with pool.gpu_lock()`의 `finally`가 실행되어 semaphore는 해제됩니다.
- 하지만 M1a에서 이미 알려진 것처럼 `ThreadPoolExecutor`에서 실행 중인 동기 변환은 실질적으로 취소되지 않고 계속 돌 수 있습니다.
- 결과적으로 첫 GPU 변환이 orphan 상태로 계속 실행되는 동안 semaphore가 풀리고, 두 번째 GPU 요청이 들어와 동시에 실행될 수 있습니다. 이는 M2 설계의 핵심인 "동시에 GPU 모델 1개만 살아있음 = VRAM 안전"을 disconnect 경로에서 깨뜨립니다.

왜 차단인가:
- M1a의 "disconnect 후 orphan compute"는 CPU/Text 경로에서는 비용 문제로 defer 가능했습니다.
- M2a의 GPU plumbing은 바로 이 semaphore로 M2b Marker의 VRAM 직렬화를 보장하려는 배관입니다. disconnect 경로에서 semaphore가 작업 생명주기보다 먼저 풀리면 GPU 배관의 안전 invariant가 성립하지 않습니다.

권고 조치:
- GPU 경로에서는 스트림 generator가 취소되더라도, executor task가 끝날 때까지 semaphore를 잡고 있어야 합니다.
- 한 가지 방식은 GPU branch/helper가 task handle을 소유하고, `finally`에서 task가 생성되었고 아직 끝나지 않았으면 `await asyncio.shield(task)` 또는 별도 drain/wait 정책으로 완료를 기다린 뒤 lock을 해제하는 것입니다. 이때 클라이언트에는 더 이상 terminal event를 보낼 수 없지만 GPU 직렬화는 유지됩니다.
- 더 깔끔하게는 `_run_conversion_stream()`을 "task 생성/결과 대기"와 "SSE yield"로 쪼개 GPU lock scope가 executor task lifetime에 정확히 붙도록 만드는 편이 좋습니다.
- 회귀 테스트는 직접 async generator를 닫거나 ASGI disconnect를 재현해서, 첫 fake GPU converter가 아직 running인 동안 두 번째 GPU converter가 진입하지 못함을 concurrent counter/barrier로 검증해야 합니다. 현재 `error_releases_lock` 테스트는 converter가 이미 끝난 뒤의 release만 확인하므로 이 경로를 잡지 못합니다.

## Recommendations

### 1. GPU cached-hit 회귀 테스트를 추가하면 분기 순서가 더 직접적으로 고정됩니다

위치:
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:211`
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:216`
- `/home/restful3/workspace/mdflow/tests/api/test_convert.py:291`

코드는 현재 올바릅니다. `lr.cached is not None` 분기가 GPU 분기보다 먼저 return하므로, 캐시 적중은 semaphore 상태와 무관하게 `cached -> done`으로 끝납니다.

다만 이번 M2a의 핵심 계약 중 하나라 테스트도 있으면 좋습니다. fake GPU converter로 한 번 캐시를 채운 뒤, semaphore를 수동으로 점유한 상태에서 같은 요청을 보내 `queued`가 없고 `cached -> done`만 나오는지 확인하면 됩니다.

### 2. fake GPU 테스트가 "lock 내부에서 변환이 실행됨"을 직접 검증하면 더 강합니다

위치:
- `/home/restful3/workspace/mdflow/tests/api/test_convert.py:316`
- `/home/restful3/workspace/mdflow/tests/api/test_convert.py:331`

현재 happy-path fake GPU 테스트는 `started.gpu == True`와 output을 확인하고, busy test는 외부에서 semaphore를 잡아 `queued`를 확인합니다. 의미는 있지만, converter 실행 시점에 lock이 실제로 held인지 직접 관찰하지는 않습니다.

후속으로 fake converter가 `app.state.pool.gpu_semaphore.locked()` 또는 shared concurrency counter를 기록하게 해 "GPU converter body는 lock 안에서만 실행된다"를 고정하면 M2b 전 안전망이 더 좋아집니다.

### 3. `queued.position = 1` 하드코딩은 M2a 단일 semaphore 기준으로는 타당합니다

위치:
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:218`
- `/home/restful3/workspace/mdflow/tests/api/test_convert.py:353`

현재 queue length를 추적하지 않는 단일 semaphore 모델에서는 `position=1`을 "GPU busy로 대기 중" 정도의 coarse signal로 쓰는 것이 합리적입니다. 정확한 대기열 순번을 의미하게 만들려면 별도 counter/queue가 필요하므로, 지금 단계에서는 reason만 강하게 의존하고 position은 best-effort로 문서화하는 정도면 충분합니다.

추출된 helper는 miss 경로의 `started -> progress* -> done|error`를 그대로 보존합니다. `MdflowError`는 원래 code/message/retryable로 SSE `error`가 되고, raw `Exception`은 `logger.exception(...)` 후 generic `INTERNAL` error event가 됩니다. CPU branch와 GPU branch 모두 같은 helper를 통과하므로 에러 매핑이 갈라지지 않습니다.

### 2. `queued` ordering은 정상 경로에서 요구사항과 맞습니다

위치:
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:216`
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:217`
- `/home/restful3/workspace/mdflow/src/mdflow/api/convert.py:219`

`queued`는 `pool.gpu_semaphore.locked()`일 때만, `async with pool.gpu_lock()` 및 `_run_conversion_stream()`의 `started` 전에 1회 방출됩니다. 세마포어가 비어 있으면 `queued` 없이 바로 lock을 잡고 `started`로 갑니다.

Python event loop 관점에서도 `locked()` check와 즉시 acquire 사이에는 uncontended path에서 실질적인 suspension point가 없어, 같은 loop의 두 요청이 둘 다 "not locked"로 통과하는 형태의 race는 잘 보이지 않습니다.

## 결론

PDF CPU converter 자체와 M1a 에러 계약 보존, cached-hit 분기 위치, `queued -> started` ordering은 전반적으로 맞습니다. 다만 GPU semaphore가 client disconnect 시 in-flight executor task보다 먼저 해제될 수 있는 점은 M2 GPU 배관의 핵심 안전성을 깨는 문제라 M2b Marker 통합 전에 반드시 막아야 합니다.
