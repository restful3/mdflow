범위: `git diff v0.4.0-m3b..HEAD`

- 로컬 설치 FastMCP 확인: `fastmcp 3.3.1`, `FastMCP.http_app(...)`, `Context.report_progress(...)`

## Blocking

### 1. `/mcp` HTTP transport가 `convert_file(path=...)`를 통해 서버 로컬 파일 읽기를 노출함

근거:
- `src/mdflow/mcp/tools.py:60-71`는 `content_base64` 또는 `path` 중 하나를 허용하고, `Path(path).read_bytes()`로 mdflow 서버 프로세스의 파일시스템을 직접 읽습니다.
- `src/mdflow/mcp/tools.py:77-78`는 클라이언트가 준 `filename`을 `filename_hint`로 넘기므로, 예를 들어 `filename="x.txt"`와 서버 파일 경로를 조합하면 텍스트 컨버터 경로로 결과가 그대로 반환될 수 있습니다.
- `src/mdflow/api/app.py:65-81`는 동일 tool set을 FastAPI 앱의 `/mcp`에 마운트합니다.
- 설계 문서도 `docs/specs/2026-05-22-m4-mcp-design.md:37-39`에서 MCP HTTP 무인증과 HTTP mode의 `path` 서버 파일시스템 접근을 인정하고 있습니다.

영향:
- stdio에서는 `path` 입력이 로컬 MCP 클라이언트 편의 기능일 수 있지만, Streamable HTTP에 동일 tool을 그대로 노출하면 HTTP 접근자가 서버의 임의 로컬 파일을 읽는 기능이 됩니다.
- `MDFLOW_MAX_INPUT_MB`는 크기 제한일 뿐 기밀성 보호가 아닙니다. 기존 `/convert` HTTP 표면은 multipart bytes 또는 URL만 받았고 서버 로컬 경로를 읽지 않았으므로, M4에서 외부 HTTP 공격 표면이 커집니다.

권장 수정:
- HTTP mount에서 `path` 입력을 비활성화하거나, stdio용 `convert_file`과 HTTP용 `convert_file`을 분리하세요.
- 최소 대안은 기본 deny로 두고 명시적 설정(예: `MDFLOW_MCP_ALLOW_PATHS`)과 allowlist root 아래의 정규화된 경로만 허용하는 방식입니다.
- 인증이 v1 비목표라면, HTTP `/mcp`에서 `path`는 비목표로 두는 편이 안전합니다. `content_base64`와 `convert_url`만으로 HTTP 사용 사례는 충족됩니다.

## Recommend

### 2. MCP 변환 작업이 HTTP의 bounded `ConcurrencyPool`을 우회함

근거:
- MCP `_run`은 `loop.run_in_executor(None, ...)`를 사용합니다: `src/mdflow/mcp/tools.py:47-48`.
- HTTP `/convert`는 fetch/lookup/run을 `pool.cpu_executor`로 제한합니다: `src/mdflow/api/convert.py:146-147`, `src/mdflow/api/convert.py:187`, `src/mdflow/api/convert.py:212-214`.
- `build_mcp()` runtime에는 pool이 없습니다: `src/mdflow/mcp/server.py:33-43`.

영향:
- `/mcp`로 들어온 장시간 변환은 event loop default executor를 사용해서 HTTP `/convert`와 다른 동시성/자원 정책을 가집니다.
- 기능 정확성 문제는 아니지만, 운영 환경에서는 `/convert`의 worker 제한을 우회하는 별도 부하 경로가 됩니다.

권장:
- M4에서 반드시 막을 필요는 없지만, HTTP-mounted MCP에는 shared pool 또는 MCP 전용 bounded executor를 넣는 것이 좋습니다.
- stdio는 default executor를 유지하더라도, `/mcp` app 생성 시 runtime에 executor를 주입할 수 있는 seam을 두면 됩니다.

### 3. `path` read와 `get_cached` disk read가 event loop에서 동기 실행됨

근거:
- `Path(path).read_bytes()`가 tool coroutine 본문에서 직접 실행됩니다: `src/mdflow/mcp/tools.py:68-71`.
- `runtime.cache.read(sha256)`도 async tool 본문에서 직접 실행됩니다: `src/mdflow/mcp/tools.py:105-112`.

영향:
- 큰 파일, 느린 filesystem, cache corruption 처리 등에서 MCP event loop가 잠깐 막힐 수 있습니다.
- 변환 본체는 executor로 잘 offload했지만, 입력 파일 read와 cache read도 I/O이므로 같은 원칙을 적용하는 편이 일관됩니다.

권장:
- `path` read는 크기 제한을 적용하면서 executor에서 수행하거나, HTTP transport에서는 위 blocking 항목처럼 아예 비활성화하세요.
- `get_cached`도 executor로 넘기면 큰 결과 파일 read가 다른 MCP 요청을 막는 일을 줄일 수 있습니다.

### 4. progress bridge는 thread-safe하나, future exception 관찰이 없음

근거:
- worker thread에서 `asyncio.run_coroutine_threadsafe(ctx.report_progress(...), loop)`를 호출합니다: `src/mdflow/mcp/tools.py:43-45`.
- FastMCP 3.3.1의 `Context.report_progress`는 progressToken이 없으면 no-op으로 돌아가므로 "토큰 없음" 케이스는 안전합니다.

판단:
- thread -> loop 마샬링 방식은 적절합니다.
- progress가 best-effort라서 worker가 Future를 기다리지 않는 것도 합리적입니다.

권장:
- `Future`에 done callback을 붙여 `future.exception()`을 소비하고 debug log만 남기면, 세션 종료 등으로 progress 전송이 실패해도 진단 가능성이 좋아집니다.
- 변환 결과를 실패시키지는 않는 현재 계약을 유지하면 됩니다.

### 5. MCP HTTP mount 테스트가 protocol-level tool call까지 검증하지 않음

근거:
- 현재 smoke는 route 존재와 bare GET이 404가 아닌지만 봅니다: `tests/mcp/test_http_mount.py:6-14`.
- tool 로직은 in-memory `Client(mcp)`로 잘 커버합니다: `tests/mcp/test_tools.py:16-126`.

권장:
- 최소 한 개의 Streamable HTTP client 초기화 및 `list_tools` 또는 `list_formats` 호출을 추가하면 lifespan/session-manager 결합 회귀를 더 직접적으로 잡을 수 있습니다.
- 또한 `convert_url`의 MCP 표면에서 실제 URL policy error가 `ToolError("[URL_*] ...")`로 매핑되는 테스트를 하나 추가하면 좋습니다. 현재 URL tool happy path는 `convert_from_url` monkeypatch라 policy 공유 자체는 정적 검토에 의존합니다.

## Notes

- 에러 계약은 전반적으로 맞습니다. `_run`의 `MdflowError -> ToolError("[CODE] message")` 매핑은 `src/mdflow/mcp/tools.py:47-50`, converter non-`MdflowError` wrap은 `src/mdflow/core/service.py:93-97`에서 성립합니다.
- `convert_file`의 exactly-one 검증, base64 decode 실패, path `OSError`, `MDFLOW_MAX_INPUT_MB` cap은 `src/mdflow/mcp/tools.py:60-76`에 구현돼 있습니다.
- `get_cached`의 sha 검증은 별도 helper가 아니라 `Cache.read()`의 `_validate_sha`에 위임하고 `ValueError -> ToolError`로 매핑합니다: `src/mdflow/core/cache.py:36-52`, `src/mdflow/mcp/tools.py:105-112`. 중복 검증 없이 충분합니다.
- 공유 composition은 의도대로 동작합니다. `build_registry()` 등록 순서는 기존 HTTP lifespan의 인라인 순서와 같고, HTTP와 MCP가 같은 함수에 의존합니다: `src/mdflow/runtime/composition.py:23-34`, `src/mdflow/api/app.py:39`, `src/mdflow/mcp/server.py:35`.
- `url_policy_from_settings()` 이동도 기존 설정 매핑과 일치합니다: `src/mdflow/runtime/composition.py:37-49`, `src/mdflow/api/app.py:52`, `src/mdflow/mcp/server.py:42`.
- `/mcp` 경로 배선은 현재 구현이 맞습니다. `build_mcp().http_app(path="/")`를 `app.mount("/mcp", mcp_app)`로 붙이면 `/mcp`와 `/mcp/`가 MCP handler에 도달합니다. 설계 문서의 `http_app(path="/mcp")` 표현(`docs/specs/2026-05-22-m4-mcp-design.md:29`, `:226`)은 현재 구현과 다르므로 문서만 정리하면 됩니다.
- FastMCP lifespan 결합은 FastMCP 3.3.1 구현 기준으로 타당합니다. mounted sub-app lifespan을 직접 진입해야 session manager가 초기화되며, `src/mdflow/api/app.py:67-70`에서 결합하고 있습니다.
- `convert_url`은 `runtime.url_policy`와 `convert_from_url()`를 사용하므로 HTTP와 같은 URL 검증/SSRF 정책을 공유합니다: `src/mdflow/mcp/tools.py:86-95`, `src/mdflow/core/url_pipeline.py:55-72`.
- `convert_url` metadata 합성은 top-level `input_kind=url`을 붙입니다: `src/mdflow/mcp/tools.py:97-99`. 단, `UrlConvertResponse.fetch` 자체도 `input_kind`를 포함합니다(`src/mdflow/core/url_pipeline.py:39-52`). HTTP `/convert`의 `fetch` dict에는 nested `input_kind`가 없으므로 완전한 shape parity가 필요하면 정리 대상입니다.
- GPU 비범위 결정은 현재 코드 기준 타당합니다. 등록된 실제 converter들은 모두 `requires_gpu=False`이고, GPU converter 추가 시 SSE와 MCP 게이팅을 함께 설계한다는 결정은 YAGNI에 부합합니다.
